## E1 — Steg Number System

**Goal:** Give users a safe, international, realistic-looking steganographic telephone number (steg number) to use as their shared channel key — either generated by the system or chosen themselves.

---

**US-E1-01 — Generate a steg number with country selection**

> As a user setting up a private channel,
> I want to select a country and generate a random steg number from that country,
> so that I get a plausible-looking number for my region.

**Acceptance criteria:**
- A country selector dropdown lists all 227 supported countries (from `phone-number-generator-js` `CountryNames` enum)
- An "Any country (random)" option is the default selection
- The generated number is in valid E.164 format (`+[CC][number]`)
- The subscriber number length is correct for the selected country
- The number is displayed in E.164 canonical form
- The number is automatically copied to clipboard on generation
- A "Generate another" button produces a different number
- Generation does not require a network request (client-side via `phone-number-generator-js` npm package)

---

**US-E1-02 — Copy generated steg number to clipboard**

> As a user who has generated a steg number,
> I want to copy it to my clipboard with one tap,
> so that I can quickly share it with the other person or save it to contacts.

**Acceptance criteria:**
- A "Copy" button appears next to the generated number
- Tapping it copies the E.164 canonical form to the clipboard
- A confirmation indicator ("Copied") appears for 2 seconds
- Works on both mobile and desktop browsers
- Falls back gracefully if clipboard API is unavailable

---

**US-E1-03 — Open channel with generated number**

> As a user who has generated a steg number,
> I want to proceed directly to the chat with the number pre-populated,
> so that I don't have to re-type it manually.

**Acceptance criteria:**
- An "Open channel with this number" button appears after number generation
- A warning card instructs the user to save the number in contacts before proceeding — "Once you leave this page, the number cannot be recovered"
- Clicking the button navigates to `/chat?phone=<e164>`
- The `/chat` entry screen shows the phone field pre-populated and read-only (locked with a lock icon)
- The user only needs to enter their PIN to open the channel
- When leaving or clearing the chat session, the phone field becomes editable again

---

**US-E1-04 — Custom number entry (removed)**

> Custom number entry and availability checking have been removed from the product. Users must use the system-generated number. This eliminates the risk of choosing guessable numbers and simplifies the flow.

---

**US-E1-05 — View the hidden-in-plain-sight setup guide**

> As a user who has chosen a steg number,
> I want clear instructions on how to save it in my phonebook,
> so that I can find it easily in the future without memorising it.

**Acceptance criteria:**
- Instructions are always visible on the `/steg-number` page below the generator card
- Step 1: open contacts app, find the other person's contact
- Step 2: add the steg number as an additional phone number entry
- Step 3: both people do this in each other's contacts
- The "Open channel with this number" button (US-E1-03) replaces the former "Done — go to chat" link

---

**US-E1-06 — Access the number generator as a standalone page**

> As a user in the initial setup phase,
> I want to use the number generator without entering the chat,
> so that I can generate and share a number before I'm ready to start chatting.

**Acceptance criteria:**
- `/steg-number` is a publicly accessible page with no authentication required
- Page passes the Passcode Test — nothing reveals the purpose to a casual observer beyond what is written on the page
- Page includes a country selector dropdown and generate button
- Page includes the setup guide (US-E1-05)
- Page includes the "Open channel with this number" flow (US-E1-03)

## E2 — Room Access & Authentication

**Goal:** Let two users independently join the same room using only a shared steg number and their own personal PIN, with no account creation and no shared secrets beyond the phone number.

---

**US-E2-01 — Enter a room with steg number and PIN**

> As a user who has agreed on a steg number with someone,
> I want to enter that number and my personal PIN to open our shared room,
> so that I can start or resume our private channel.

**Acceptance criteria:**
- Entry screen shows two fields: phone number (masked by default) and PIN (numeric, masked)
- Submitting triggers client-side derivation: `room_hash` and `access_hash`
- Server verifies access and returns `room_id` on success
- Client derives `enc_key` and `sender_hash` locally using `room_id`
- Successful entry opens the chat screen
- Failed entry shows a neutral error: "Could not open this room"
- The error message does not distinguish between wrong number, wrong PIN, or non-existent room

---

**US-E2-02 — Use a different PIN for each room**

> As a security-conscious user,
> I want to set a unique PIN for each of my conversations,
> so that if one conversation's PIN is somehow compromised, my other conversations remain protected.

**Acceptance criteria:**
- The app does not enforce or store PINs — each room accepts any PIN on first join
- The first `access_hash` presented for a given `room_hash` is stored as that user's access record
- A different PIN for a different room is handled identically — the app has no concept of "a user's PIN"
- The onboarding guide explains both the same-PIN and unique-PIN strategies and their tradeoffs
- No PIN management interface exists — PINs are never stored anywhere by the app

---

**US-E2-03 — Use the same PIN across multiple rooms**

> As a user who values simplicity,
> I want to use the same PIN for all my conversations,
> so that I only have one thing to remember.

**Acceptance criteria:**
- The same PIN works for any room the user has joined with it
- No app-level restriction prevents this
- Onboarding acknowledges this is a valid choice with a brief note on the tradeoff
- Functionally identical to US-E2-02 from the system's perspective

---

**US-E2-04 — Receive a neutral error after 10 failed PIN attempts**

> As the system,
> I want to lock a user out after 10 failed access attempts,
> so that brute-force PIN attacks are impractical.

**Acceptance criteria:**
- After each failed attempt, a counter is shown: "X attempts remaining"
- After 10 failures, the server sets `locked_until = now + 30 minutes` on the access record
- During lockout, any access attempt with that `room_hash + access_hash` returns a lockout response
- The error shown during lockout: "Too many failed attempts. Try again in 30 minutes." — no countdown
- After lockout expires, the counter resets and access is permitted normally
- The lockout is per `room_hash + access_hash` — it does not affect the room for other users

---

**US-E2-05 — First user to join creates the room**

> As the first person to enter a new steg number on sTELgano,
> I want the room to be created automatically on my first join,
> so that I don't need a separate "create room" step.

**Acceptance criteria:**
- If no room with the derived `room_hash` exists, the server creates it and the access record in one transaction
- The creating user is in the room immediately — no confirmation step
- The room starts in empty state — no messages
- The second user joining (with same number, different PIN) creates their own access record for the same room
- No "waiting for other user to join" state is shown — the room is ready immediately

---

**US-E2-06 — Toggle phone number visibility on entry screen**

> As a user entering my steg number,
> I want to be able to reveal the masked field to verify what I've typed,
> so that I can correct typos before submitting.

**Acceptance criteria:**
- Phone number field is `type="password"` by default
- An eye icon button toggles between masked and visible
- PIN field remains always masked
- The toggle does not submit the form
- Icon changes between eye-open and eye-closed states

---

**US-E2-07 — See a loading indicator during key derivation**

> As a user who has submitted the entry form,
> I want to see a loading state while the encryption key is being derived,
> so that I understand why there is a brief pause and don't assume the app has frozen.

**Acceptance criteria:**
- As soon as the form is submitted, the Enter button shows a loading state ("Verifying…" + spinner)
- The loading state persists for the duration of PBKDF2 key derivation (~1.5–2.5s on mobile)
- After derivation completes, the server request is made
- If the server responds with success, the chat screen loads
- If the server responds with failure, the error is shown and the button returns to "Enter"

---

**US-E2-08 — Entry screen passes the Passcode Test**

> As a user whose phone is picked up by a suspicious person,
> I want the entry screen to reveal nothing about the purpose of the app,
> so that a casual observer cannot determine that a private channel exists.

**Acceptance criteria:**
- No recent conversation history is displayed
- No contact names or identifiers are visible
- Field labels are minimal and generic
- No "conversation with X" or timestamp information is shown
- Error messages do not confirm the existence of any room
- The only URL in browser history is `stelgano.com` or `stelgano.com`

## E3 — N=1 Async Messaging

**Goal:** Implement the N=1 invariant — at most one message exists per room at any time — with asynchronous delivery so both users do not need to be online simultaneously.

---

**US-E3-01 — Send an encrypted message**

> As a user in a room where it is my turn to speak,
> I want to type a message and send it,
> so that the other person receives my encrypted message when they next open the app.

**Acceptance criteria:**
- Client encrypts plaintext using AES-256-GCM with the locally-derived `enc_key`
- Encrypted ciphertext + 96-bit nonce sent to server via Phoenix Channel
- Server verifies the room is active and it is this user's turn (no existing undeleted message from this user)
- Server stores `(room_id, sender_hash, ciphertext, iv)`
- Server broadcasts `new_message` to all channel members
- Sender sees their message appear in the chat window (right-aligned bubble)
- Sender's input disappears and is replaced by "Waiting for reply…" animation
- Sender receives single-tick ✓ confirmation

---

**US-E3-02 — Receive a message sent while offline**

> As a user returning to a conversation after being away,
> I want to see any message the other person sent in my absence,
> so that I can read and reply when I'm ready.

**Acceptance criteria:**
- On joining the channel, server sends `current_message` in the join response payload if one exists
- Client decrypts the message using `enc_key` derived from the same inputs
- Message is displayed correctly (left-aligned received bubble if sender_hash ≠ mine)
- If no message exists, the room is blank — no placeholder text
- The read receipt is fired when the user's browser confirms the message is in view

---

**US-E3-03 — Reply to a message**

> As a user who has received a message,
> I want to type and send a reply,
> so that the previous message is permanently deleted and my reply becomes the current message.

**Acceptance criteria:**
- Replying is only possible when a received message is visible and it is the user's turn
- Client encrypts reply and sends via channel
- Server atomically: (1) soft-deletes all existing messages for this room, (2) inserts the new message — in a single transaction
- N=1 invariant is maintained — no moment exists where two messages are active
- Sender sees their reply appear (right-aligned bubble)
- The previous message disappears from both screens simultaneously
- Replying user's input locks; counterparty's input activates

---

**US-E3-04 — View a single-tick confirmation after sending**

> As a user who has sent a message,
> I want to see a single-tick indicator,
> so that I know my message was received and stored by the server.

**Acceptance criteria:**
- Single tick ✓ appears on the sent bubble immediately after the server acknowledges the message
- Tick is styled in a muted colour (not full accent — it's not yet been read)
- Tick does not appear until the channel `push` receives an `:ok` reply

---

**US-E3-05 — See a double-tick when the other person reads my message**

> As a user waiting for the other person to read my message,
> I want the tick to update to a double-tick,
> so that I know the message has been seen.

**Acceptance criteria:**
- Server sets `read_at` on the message when the recipient's IntersectionObserver dwell fires (500ms)
- Server broadcasts `message_read` to all channel members
- Sender's single tick ✓ updates to double tick ✓✓ in real time
- If the sender is offline when read occurs, they see ✓✓ the next time they open the room

---

**US-E3-06 — Edit a sent message before it is read**

> As a user who has sent a message with a typo or change of mind,
> I want to edit it before the other person reads it,
> so that they see the corrected version.

**Acceptance criteria:**
- Edit option is available (long-press mobile / right-click desktop) only while `read_at` is null
- The current bubble text is populated into the input field for editing
- Submitting the edit sends new encrypted ciphertext to the server
- Server updates the stored message and broadcasts `message_edited`
- Both screens display the updated text
- An "edited" label appears beneath the bubble
- If `read_at` is set at the moment of edit submission, the server rejects the edit with `:already_read`

---

**US-E3-07 — Delete a sent message before it is read**

> As a user who has sent a message I want to retract,
> I want to delete it before the other person reads it,
> so that the message never reaches them.

**Acceptance criteria:**
- Delete option is available only while `read_at` is null
- A single confirmation prompt: "Delete this message?" with "Delete" and "Cancel"
- On confirm: server hard-deletes the message (not soft-delete — removes from DB entirely)
- Server broadcasts `message_deleted` to all channel members
- Both screens clear the message and return to empty room state
- Both users' inputs become active
- If `read_at` is set at the moment of deletion, the server rejects with `:already_read`

---

**US-E3-08 — Cannot send when it is not my turn**

> As a user who has already sent a message awaiting a reply,
> I want to be prevented from sending another message,
> so that the turn-based conversation rhythm is preserved.

**Acceptance criteria:**
- After sending, the input area is replaced by the "Waiting for reply…" animation
- No send mechanism is available — no input, no button
- If the user somehow constructs a channel push, the server rejects it: `{:error, :not_your_turn}`
- The input only reappears after the other party sends a reply (N=1 swap)
- This state persists correctly across multi-device sessions (same lock state on all devices)

---

**US-E3-09 — See a typing indicator while the other person is composing**

> As a user waiting for the other person's reply,
> I want to see a typing indicator when they are composing,
> so that I know they are actively responding.

**Acceptance criteria:**
- A `typing` event is pushed to the channel on each keystroke in the input
- The counterparty sees a three-dot animated indicator above the message area
- The indicator disappears after 3 seconds of no keystroke (debounced)
- The indicator disappears immediately when the counterparty sends
- The indicator is not shown when it is not the counterparty's turn

---

**US-E3-10 — Messages are not lost if I close the app before the other person replies**

> As a user who has sent a message and closed the app,
> I want my message to still be there when the other person opens the app,
> so that the conversation is not broken by my going offline.

**Acceptance criteria:**
- Messages are persisted to the database on send — not held in memory
- Closing the browser tab or app has no effect on the server-side message
- The recipient receives the message on their next join, regardless of how much time has passed
- TTL expiry (if configured) is the only mechanism that removes an unread message over time

## E4 — Chat UI & Real-Time Features

**Goal:** Deliver a polished, intuitive chat interface that communicates the N=1 conversation model clearly and feels natural to use.

---

**US-E4-01 — See my sent messages on the right side of the chat window**

> As a user looking at the chat screen,
> I want my sent messages to appear on the right side,
> so that it is immediately clear which messages are mine.

**Acceptance criteria:**
- Messages where `sender_hash` matches the locally stored `sender_hash` are right-aligned
- Sent bubbles use emerald gradient background (`linear-gradient(135deg, #10B981, #047857)`) with white text
- Sent bubbles use the geometry: `border-radius: 1.5rem 1.5rem 0.25rem 1.5rem` (24px base, 4px tail)

---

**US-E4-02 — See received messages on the left side**

> As a user looking at the chat screen,
> I want the other person's messages to appear on the left side,
> so that the conversation direction is immediately legible.

**Acceptance criteria:**
- Messages where `sender_hash` does not match mine are left-aligned
- Received bubbles use frosted glass background (`rgba(255, 255, 255, 0.08)` with `backdrop-filter: blur(8px)`) and `--text-main` text
- Received bubbles use the geometry: `border-radius: 1.5rem 1.5rem 1.5rem 0.25rem` (24px base, 4px tail)

---

**US-E4-03 — See the "Waiting for reply" animation when it is not my turn**

> As a user who has sent a message,
> I want to see a clear animation indicating I am waiting,
> so that I understand the turn-based model without confusion.

**Acceptance criteria:**
- Three animated dots in a staggered-bounce loop are displayed
- "Waiting for reply…" label appears next to or below the dots
- The input textarea and send button are entirely absent — not disabled, absent
- The animation runs continuously until the counterparty replies
- The animation respects `prefers-reduced-motion` — static dots shown as fallback

---

**US-E4-04 — See the conversation return to empty state after a message is deleted**

> As a user who deleted their sent message (or whose counterparty deleted theirs),
> I want the chat window to return to the blank state,
> so that it is clear no message exists and both of us can speak.

**Acceptance criteria:**
- On `message_deleted` event, the bubble is removed with a subtle fade-out
- The chat window shows an empty state
- Both users' inputs become active
- No placeholder text, no "message deleted" tombstone — just empty

---

**US-E4-05 — See a smooth bubble animation when a new message arrives**

> As a user in an active conversation,
> I want messages to appear with a subtle entrance animation,
> so that the arrival of a new message is clearly communicated.

**Acceptance criteria:**
- New bubbles animate in: scale `0.95 → 1.0` + opacity `0 → 1` over 150ms
- Animation fires for both incoming and outgoing messages
- `prefers-reduced-motion` disables the animation — instant display instead

---

**US-E4-06 — Resize the message input as I type**

> As a user composing a long message,
> I want the input area to grow as I type,
> so that I can see my full message without scrolling within the input.

**Acceptance criteria:**
- Textarea starts at 1 row height
- Auto-resizes up to a maximum of ~140px (approximately 6 lines)
- Beyond 140px, the textarea scrolls internally
- Resize is smooth — no layout jumping
- Width is 100% of the available area minus the send button

---

**US-E4-07 — Send a message with the Enter key on desktop**

> As a desktop user composing a message,
> I want to press Enter to send,
> so that the experience feels natural for keyboard users.

**Acceptance criteria:**
- `Enter` alone sends the message
- `Shift+Enter` inserts a newline
- `Enter` on mobile (virtual keyboard) also sends
- An explicit send button is always available as an alternative

---

**US-E4-08 — See a character count warning approaching the limit**

> As a user composing a long message,
> I want to see a warning when I am approaching the character limit,
> so that I know to wrap up before being cut off.

**Acceptance criteria:**
- Character counter appears when the message exceeds 3,500 characters
- Counter shows: `3,847 / 4,000`
- Counter turns warning colour at 3,900 characters
- Server rejects messages over 4,000 characters
- Client validates and shows an inline error before sending if over limit

---

**US-E4-09 — See the "edited" label on a message that has been modified**

> As a recipient reading a message,
> I want to know if the message was edited after it was originally sent,
> so that I have an accurate picture of the conversation.

**Acceptance criteria:**
- An "edited" label appears beneath the bubble when `message_edited` is received
- The label is styled in `--text-muted` — present but not prominent
- The label appears on both the sender's and recipient's screens simultaneously

## E5 — Lock & Session Management

**Goal:** Protect an open chat session from anyone who finds an unlocked device, while keeping the return-to-chat experience smooth for the legitimate user.

---

**US-E5-01 — Lock the chat manually**

> As a user who needs to put down my phone mid-conversation,
> I want to lock the chat with one tap,
> so that anyone who picks up my phone cannot read the conversation.

**Acceptance criteria:**
- A lock icon is in the session header
- Tapping it immediately blurs/overlays the chat content
- The lock screen is shown with a PIN entry field
- No message content is visible through the overlay

---

**US-E5-02 — Chat locks automatically after inactivity**

> As a user who may walk away from my device,
> I want the chat to lock itself after a configurable period of inactivity,
> so that I am protected even if I forget to lock manually.

**Acceptance criteria:**
- Inactivity timeout options: `30s · 1min · 5min · 15min · 30min · never`
- Default: 5 minutes
- Inactivity is measured from last user interaction (tap, keypress, scroll)
- Tab losing focus also starts the inactivity timer
- When the timer fires, lock behaviour identical to US-E5-01
- Timeout preference is saved in `localStorage` and persists across sessions

---

**US-E5-03 — Resume chat with correct PIN**

> As a returning user whose chat has been locked,
> I want to enter my PIN to instantly resume,
> so that I can get back to the conversation without re-entering the full credentials.

**Acceptance criteria:**
- PIN field is shown on the lock screen
- Correct PIN: chat is immediately unblurred, lock screen dismissed — no network request needed
- `enc_key` is re-derived from `phone` + `room_id` (both in sessionStorage) locally
- Transition from lock to chat: smooth, < 200ms
- The phone number field does not need to be re-entered on the lock screen

---

**US-E5-04 — Receive an error for a wrong PIN on the lock screen**

> As a user who misremembers their PIN,
> I want to see a clear error and know how many attempts remain,
> so that I can try again confidently.

**Acceptance criteria:**
- Wrong PIN shows: "Incorrect PIN. X attempts remaining."
- A 2-second delay is enforced between attempts (client-side + server-side)
- After 5 wrong attempts: session is cleared, entry screen is shown
- The room is NOT expired — the conversation is still on the server

---

**US-E5-05 — Log out without expiring the room**

> As a user who wants to close the session without ending the conversation,
> I want to log out and return to the entry screen,
> so that the other person's pending message is not lost.

**Acceptance criteria:**
- A "Leave" button in the session header
- No confirmation dialog — immediate action
- sessionStorage is cleared, `enc_key` is discarded from memory
- Browser is returned to the entry screen
- Room and any current message remain on the server
- The user can rejoin by entering the steg number and PIN again

---

**US-E5-06 — Use panic clear from the lock screen**

> As a user in a situation where I need to remove all traces immediately,
> I want a single tap to clear my session completely,
> so that nothing about my conversation is accessible on this device.

**Acceptance criteria:**
- A subtle "Clear session" text link appears on the lock screen
- One tap — no confirmation
- sessionStorage is cleared immediately
- `enc_key` is discarded from memory
- Entry screen is displayed
- Room and current message on the server are NOT affected

---

**US-E5-07 — Lock screen passes the Passcode Test**

> As a user whose locked device is picked up by a suspicious person,
> I want the lock screen to reveal nothing about my conversation,
> so that a casual observer cannot determine that a conversation is in progress.

**Acceptance criteria:**
- Lock screen shows only: sTELgano wordmark, PIN field, "Enter PIN to resume" label, "Clear session" link
- No conversation partner identity visible
- No message content visible or inferrable
- No timestamps or activity indicators visible
- Chat content is fully obscured (not just scrolled out of view)

---

## E6 — Multi-Device Access

**Goal:** Allow a user to access the same conversation from different devices without any special setup or synchronisation mechanism.

---

**US-E6-01 — Access my conversation from a second device**

> As a user who switches between my laptop and my phone,
> I want to enter the same steg number and PIN on either device and see the same conversation state,
> so that I can continue chatting regardless of which device I am on.

**Acceptance criteria:**
- Entering the same credentials on a new device produces the same `room_hash` and `access_hash`
- Server returns the same `room_id` — which is used to derive the same `enc_key`
- The current message (if any) is delivered and decrypts correctly on the new device
- Input lock state (waiting / active) is correctly reflected based on `sender_hash` comparison
- No special "add device" flow is required

---

**US-E6-02 — See real-time updates on all my connected devices simultaneously**

> As a user connected on two devices at the same time,
> I want both to update in real time when the other person sends a message,
> so that I always see the latest state regardless of which device I'm looking at.

**Acceptance criteria:**
- Both devices are connected to the same Phoenix Channel topic
- A `new_message` broadcast is received by all connected members
- Both devices decrypt and display the message simultaneously
- Both devices update input state simultaneously
- Typing indicators appear on all of a user's connected devices when the counterparty types

---

**US-E6-03 — Correctly identify my sent messages as "mine" on a new device**

> As a user opening my conversation on a new device,
> I want messages I previously sent to still appear on the right side,
> so that the conversation layout makes sense.

**Acceptance criteria:**
- `sender_hash` is derived deterministically from `phone + access_hash + room_hash + SENDER_SALT` (includes access_hash for per-PIN sender isolation)
- Same phone on new device → same `sender_hash`
- Messages with matching `sender_hash` render on the right; others on the left
- The layout is correct from the first load — no mis-assignment

---

**US-E6-04 — Input remains locked on all devices after I have sent a message**

> As a user connected on two devices simultaneously after sending a message,
> I want both devices to show the "waiting for reply" state,
> so that I cannot accidentally attempt to send from either device.

**Acceptance criteria:**
- When User A sends from Device 1, `new_message` is broadcast to all channel members
- Device 2 (also User A) receives the broadcast
- Device 2 checks: `message.sender_hash === my sender_hash` → input locks on Device 2 too
- Both devices show "Waiting for reply…"
- When the counterparty replies, both devices unlock simultaneously

## E7 — Room Lifecycle & TTL

## E10 — Cryptographic Architecture

**Goal:** Implement the complete client-side cryptographic system as a single, auditable, dependency-free vanilla JS module.

---

**US-E10-01 — Implement normalise(phone)**

> As the cryptographic module,
> I need a normalise function that consistently strips formatting from phone numbers,
> so that the same number typed with different formatting always produces the same hash.

**Acceptance criteria:**
- Strips: spaces, dashes, dots, parentheses, plus signs
- Result: digits only (no country code `+` prefix — just digits)
- `normalise("+254 700 123-456") === "254700123456"`
- `normalise(normalise(x)) === normalise(x)` — idempotent
- Tested with property-based tests (StreamData)

---

**US-E10-02 — Implement roomHash(phone)**

> As the cryptographic module,
> I need a room hash function,
> so that the same phone number always produces the same opaque room identifier.

**Acceptance criteria:**
- `roomHash(phone) = SHA-256(normalise(phone) + ":" + ROOM_SALT)` (hex output)
- Uses `crypto.subtle.digest('SHA-256', ...)`
- Deterministic: same input → same output
- Tested for determinism and distinctness from accessHash

---

**US-E10-03 — Implement accessHash(phone, pin)**

> As the cryptographic module,
> I need an access hash function that incorporates the PIN,
> so that the PIN gates server access without being transmitted.

**Acceptance criteria:**
- `accessHash(phone, pin) = SHA-256(normalise(phone) + ":" + cleanPin + ":" + ACCESS_SALT)` (hex)
- `cleanPin = pin.replace(/\D/g, '')` — digits only
- `accessHash(p, "1234") !== accessHash(p, "5678")` for all `p`
- Deterministic for same inputs

---

**US-E10-04 — Implement deriveKey(phone, roomId)**

> As the cryptographic module,
> I need a key derivation function,
> so that both users independently arrive at the same AES-256-GCM encryption key.

**Acceptance criteria:**
- `deriveKey(phone, roomId) = PBKDF2(normalise(phone), roomId + ENC_SALT, 600_000, SHA-256, 256 bits)`
- Returns a non-extractable `CryptoKey` object for AES-256-GCM
- Same phone + same roomId → identical key on every call
- Different phone or different roomId → different key
- Loading indicator is shown in the UI during derivation

---

**US-E10-05 — Implement encrypt(plaintext, key) and decrypt(ciphertext, iv, key)**

> As the cryptographic module,
> I need encrypt and decrypt functions,
> so that messages can be sent and received as opaque ciphertext.

**Acceptance criteria:**
- `encrypt`: generates 96-bit random IV via `crypto.getRandomValues`; uses `AES-GCM` encrypt; returns `{ciphertextB64, ivB64}`
- `decrypt`: decodes base64; uses `AES-GCM` decrypt with provided IV; returns plaintext string
- `decrypt(encrypt(plaintext, key).ciphertextB64, encrypt(plaintext, key).ivB64, key) === plaintext`
- Tampered ciphertext causes `decrypt` to throw (GCM auth tag failure) — tested explicitly

---

**US-E10-06 — Crypto constants are defined exactly once**

> As a developer maintaining the codebase,
> I want all cryptographic constants to exist in a single file,
> so that there is no possibility of drift between the client JS and the security documentation.

**Acceptance criteria:**
- `ROOM_SALT`, `ACCESS_SALT`, `SENDER_SALT`, `ENC_SALT`, `PBKDF2_ITER` defined only in `assets/js/crypto/anon.js`
- No other file in the repository defines or duplicates these constants
- The security page (`/security`) links directly to this file on GitHub
- A comment in the file notes that changing any constant is a breaking change requiring all rooms to be recreated

## E17 — Room Lifecycle & Cleanup

**Goal:** Keep the database lean and the steg number pool healthy. Every room that is no longer in active use should be automatically detected and hard-deleted, with its steg number returned to the pool. Clean rooms = clean pool = zero bloat.

---

**US-E17-01 — Auto-delete rooms where second party never joined**

> As the system,
> I want to delete rooms where only one party has ever joined,
> so that abandoned rooms do not occupy steg numbers indefinitely.

**Acceptance criteria:**
- `peer_joined_at` column tracks the timestamp of the second distinct access_hash joining a room
- Oban job `RoomPurge` runs every 15 minutes
- Rooms where `peer_joined_at IS NULL AND inserted_at < NOW() - INTERVAL '24 hours'` are hard-deleted
- On deletion: room record, all room_access records, and messages (if any) are hard-deleted in a single transaction
- room_hash removed from rooms table — steg number immediately available for reuse
- No broadcast sent (no second party was ever connected)
- Deletion logged as `cleanup_reason: no_peer` in aggregate metrics (no room identifier retained)

---

**US-E17-02 — Auto-delete stale rooms with no messages**

> As the system,
> I want to delete rooms where both parties joined but never exchanged a message,
> so that stale "almost-used" rooms do not accumulate.

**Acceptance criteria:**
- `first_message_at` column tracks timestamp of first message sent in a room
- Rooms where `first_message_at IS NULL AND peer_joined_at < NOW() - INTERVAL '48 hours'` are hard-deleted
- Both `peer_joined_at` and `first_message_at` columns indexed for efficient batch queries
- Deletion logged as `cleanup_reason: no_messages` in aggregate metrics
- On deletion: full hard delete of room + all access records

---

**US-E17-03 — Auto-expire rooms with unreplied messages at TTL**

> As the system,
> I want to expire rooms where a message was sent but never replied to once the TTL elapses,
> so that one-sided conversations do not hold steg numbers open indefinitely.

**Acceptance criteria:**
- Rooms where `ttl_expires_at < NOW() AND is_active = TRUE` are expired regardless of message state
- If an active message exists (`read_at IS NULL`), it is hard-deleted atomically with the room
- `room_expired` broadcast sent to any connected WebSocket clients
- Connected clients see the expiry screen; sender learns the channel closed
- Deletion logged as `cleanup_reason: ttl_elapsed`
- Steg number immediately returned to pool

---

**US-E17-04 — Handle dedicated number subscription lapse**

> As the system,
> I want to give dedicated number holders a 7-day grace period when their subscription lapses,
> so that a missed renewal does not immediately destroy an active channel.

**Acceptance criteria:**
- When Stripe webhook fires `customer.subscription.deleted`, `is_active` is NOT immediately set to false
- A `grace_expires_at = NOW() + 7 days` timestamp is set on the room
- In-chat warning shown immediately: "Your dedicated number subscription has lapsed. Renew within 7 days to keep this number."
- Renewal link shown in the expiry warning; clicking it redirects to Stripe Customer Portal
- If renewed within grace: `grace_expires_at` cleared; room continues normally
- If not renewed: room hard-deleted at `grace_expires_at`; steg number returned to shared pool
- Stripe sends renewal reminder emails at 30 days, 7 days, and 1 day before lapse (configured in Stripe)

---

**US-E17-05 — Display aggregate pool hygiene metrics in admin**

> As an admin,
> I want to see how many rooms were cleaned up today and by what trigger,
> so that I can monitor whether the cleanup system is working correctly.

**Acceptance criteria:**
- Admin dashboard shows: rooms cleaned today (total), breakdown by cleanup_reason
- `metric_snapshots` table records hourly cleanup counts by reason (no room identifiers)
- Admin NL → SQL query interface can answer: "How many rooms were cleaned up in the last 7 days by reason?"
- No room_hash or steg number appears in any cleanup metric

---

**US-E17-06 — Notify connected parties when a room is purged mid-session**

> As a user in an active chat when the server purges a room,
> I want to see the expiry screen immediately,
> so that I am not left staring at a broken interface.

**Acceptance criteria:**
