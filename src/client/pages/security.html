Security Architecture

Security & Privacy.

sTELgano-std-1 · Document Reference AE-2026

Our Promise

Your messages are encrypted on your device before they ever leave your browser. The server never sees your phone number, your PIN, or the content of your messages. It only receives opaque hashes and ciphertext that are meaningless without your credentials.

There are no accounts to create, no email addresses to provide, and no profile data to fill in. You share a number with someone, each of you picks a PIN, and that is the entire setup. We built sTELgano on a zero-knowledge architecture: we cannot read your messages, we cannot identify you, and we cannot recover your data. That is not a limitation — it is the design.

No metadata is collected. No chat history is retained. No message logs exist on the server. When a reply is sent, the previous message is atomically deleted. At most one message exists in any room at any time.

How Encryption Works

Derivation Chain

Every piece of data the server stores is derived from your shared number and PIN through a series of one-way cryptographic functions. Here is what happens when you enter your credentials:

  1. Your phone number is normalised to E.164 format — spaces, dashes, and formatting are stripped.
  2. A room hash is computed so the server can look up your room without knowing your number.
  3. An access hash is computed from your number and PIN — this proves you know the PIN without revealing it.
  4. An encryption key is derived using 600,000 rounds of PBKDF2 — this key never leaves your browser.
  5. A sender hash is computed so the server can tell which participant sent a message, without knowing who they are.
phone       = normalise(phone_number)
            // Strip spaces, dashes, formatting → E.164 standard

# Server-side identifiers (opaque hashes)
room_hash   = SHA-256(phone + ":" + ROOM_SALT)
            // Unique room ID — cannot be reversed to recover the phone number

access_hash = SHA-256(phone + ":" + PIN + ":" + ACCESS_SALT)
            // Proves PIN knowledge without revealing the PIN itself

# Client-side secrets (never leave the browser)
enc_key     = PBKDF2(phone, room_id + ENC_SALT, 600000, SHA-256, 256-bit)
            // Master encryption key — 600K iterations per OWASP 2023

sender_hash = SHA-256(phone + ":" + access_hash + ":" + room_hash + ":" + SENDER_SALT)
            // Unique sender identity per room session

Encryption Specifications

Cipher
AES-256-GCM

Authenticated encryption with associated data. Industry standard for confidentiality and integrity.

Key Derivation
PBKDF2 600K

600,000 iterations per OWASP 2023 recommendation. Approximately 2 seconds of computation per derivation.

Hash Function
SHA-256

Used for all identifier hashes. Produces 256-bit digests that are computationally irreversible.

Nonce
96-bit Random

Cryptographically random initialisation vector generated fresh for every message encryption.

Auth Tag
128-bit MAC

Message authentication code that detects any tampering with the ciphertext.

Key Length
256-bit

Full 256-bit key space. Brute-forcing this would take longer than the age of the universe.

Threat Model

What we protect against

  • Casual device snooping and shoulder surfing
  • ISP and network operator packet inspection
  • Server-side database breaches
  • Browser history and cache forensics
  • Device seizure — no app to find, no chat history to recover
  • Intimate-access attackers (partner, family member with device access)

Out-of-Scope Risks

We are honest about our limits. For maximum protection, always use sTELgano in Incognito/Private mode. sTELgano does not protect against:

  • Nation-state surveillance infrastructure
  • Government subpoenas and law enforcement
  • Targeted endpoint malware and keyloggers
  • Physical coercion or forced PIN disclosure
  • A compromised browser or operating system
  • Network observers who see that you connect to stelgano.com (SNI / DNS / IP)

What the Network Sees

TLS protects the contents of what you send us — never the fact that you're sending it. A partner on the same Wi-Fi, the coffee shop's router, your ISP, or a corporate / school network appliance can observe the following with standard tools (Wireshark, nftables, router logs) even while everything above the TCP/TLS layer stays encrypted:

  • The destination hostname — TLS 1.2 and 1.3 send the hostname (stelgano.com) in cleartext during the ClientHello (SNI). Encrypted ClientHello (ECH) is not yet widely deployed.
  • DNS lookups — unless you use DNS-over-HTTPS or DNS-over-TLS, the query for stelgano.com is also visible.
  • IP address + connection timing — someone inspecting a router log or a NetFlow export can see that your device talked to our IP for a period of time.

If that disclosure is a problem for you, use a VPN or Tor before loading sTELgano. Neither the product nor the protocol can hide the shape of your traffic; the design honestly acknowledges this rather than pretending network-layer metadata doesn't exist.

Paid Extensions: A Residual Correlation

If you pay for a dedicated number, the payment flow is deliberately structured so the extension_tokens table never stores a room_id. No DB dump can answer “which room did this payment pay for?”.

We still honestly disclose one residual weakness: a server operator with both DB read access and application log access could temporally align the payment timestamp with a room's TTL update to infer the link. We raise the cost of this correlation by (a) running token and room updates in separate transactions with a random jitter between them, and (b) rounding the stored expiry to the top of its hour so the stored value does not encode the redemption moment. We do not claim to eliminate it against a determined server operator — and this is consistent with the product's broader stance that we do not defend against law-enforcement-grade server access.

Operator Disclosure

sTELgano's cryptographic design protects you against passive compromise of the server — database dumps, backup leaks, opportunistic intrusion, and a new operator taking over the service with different intentions. What is stored is hashes and ciphertext, and those reveal nothing without credentials the server never sees.

There is one class of adversary the architecture cannot defeat on its own: an adversary who can compel the operator to serve modified JavaScript. If a subpoena, court order, or equivalent legal instrument requires the operator to push compromised client code to a specific user, the architecture has no way to stop it — the browser trusts whatever JavaScript arrives at page load.

The operator does not pre-commit to refusing such process. Unlike services that have publicly committed to shutting down rather than comply (Lavabit in 2013 being the canonical example), the operator of stelgano.com lacks the financial and legal resources to contest a lawful order. You should assume that if this service is subpoenaed, the operator will comply. If that risk is unacceptable for your use case, either self-host on infrastructure you control, or use a tool whose operator has made — and can afford to keep — a stronger commitment.

This non-commitment reflects the baseline of the privacy-tools landscape rather than an unusual position. Even Signal — with substantial foundation funding, and native-app distribution via Apple/Google that sidesteps the web bootloading problem entirely — has not made a Lavabit-style shutdown pledge; its posture is architectural minimisation (store little, so compliance yields little) rather than refusal. Lavabit's 2013 shutdown is the exception, not the baseline.

This disclosure exists because vague privacy claims create false confidence. We would rather state this limit plainly than imply a protection that only holds until the first legal letter arrives.

The Passcode Test

Design Principle

Every design decision in sTELgano must pass a single test:

“A suspicious person unlocks your phone and opens sTELgano. What do they see?”

The answer: a blank entry screen with two fields. Nothing else. No message previews, no contact list, no notification badges, no chat history. There is nothing to find because nothing is stored.

Data Lifecycle

CORE RULE

N=1 Invariant

At most one message exists per room at any time. When a reply is sent, the previous message is atomically deleted in the same transaction. This mirrors how real conversations work — people ask what their partner was replying to, quote it in their reply, or paraphrase it. N=1 enforces that rhythm rather than fighting it. If you need a local record, copy before replying.

IMMEDIATE

Immediate Hard Delete on Reply

When you reply, the previous message is permanently hard-deleted from the database in the same transaction that inserts your new message. The old row is gone instantly — no deferred cleanup, no residual data.

TTL ENFORCED

Room Expiry via TTL

Rooms self-destruct via a Durable Object alarm set at creation. When the TTL fires, the room broadcasts expiry to all connected clients and atomically wipes every access record and message from its storage. No batch job, no deferred cleanup — the alarm fires at the exact moment the TTL expires. Expired rooms cannot be rejoined.

Zero-Trace Forensics

History Masking

Browser history is a significant forensic trail. sTELgano uses Active History Masking to ensure that even if you navigate through the chat or generator, only the root path (/) is ever recorded in your browser's persistent history. Anyone inspecting your history after the session will see no record of specific feature usage.

Incognito Hardening

While we mask history, we strongly recommend using Incognito / Private Mode. This prevents the browser from caching assets, storing favicons, or recording the visit timestamp in long-term databases. Combined with our memory-only key storage, this creates a session that truly vanishes the moment the tab is closed.

Vault Isolation

To prevent accidental persistence of credentials, sTELgano implements Vault Isolation. We use non-standard field attributes and one-time-code markers to discourage browser password managers from offering to save your Secret Number or PIN. This ensures your primary authentication artifacts stay in your memory, not in a digital vault.

Known Limits

Multi-Channel Memorisation Burden

Using sTELgano with more than one person means carrying a unique steg number and PIN per channel. The correct approach — a unique PIN per channel — is also the hardest to maintain:

  • Unique PIN per channel — strongest security, but N items to memorise for N channels.
  • Same PIN everywhere on this site — a browser autofill saving that PIN would suggest it for every channel, reducing them all to the same credential.
  • PIN saved in a password manager — breaks the passcode test (the manager’s vault is a forensic trail, and the entry label names the site).

There is no fully convenient answer. This is an honest statement of the trade-off, not a gap we plan to patch.

Advanced: Derived PINs

OPTIONAL PATTERN

For users managing multiple channels, one approach reduces memorisation from N items to 2 (a global PIN + a derivation rule) while keeping each channel’s PIN unique and autofill-proof:

Pick a fixed global PIN and a rule that extracts digits from the shared steg number. Apply the rule to each number to derive that channel’s final PIN. Example rule shapes:

  • global + last 2 digits of the number
  • global + middle 3 digits (e.g. digits 4–6)
  • global + digits at a chosen position, reversed

Honest caveats: theoretical entropy is slightly lower than fully independent PINs — immaterial against the server’s 10-attempt lockout but relevant for offline brute-force on leaked ciphertext. If two channels are compromised (steg number + final PIN for both), an attacker can infer the global PIN and rule. This is consistent with the stated threat model but worth naming. The rule being shared across channels is also the load-bearing secret — writing it down removes the benefit.
