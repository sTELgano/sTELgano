What We Gave Up.

An honest accounting of the tradeoffs we made migrating sTELgano to Cloudflare Workers. The gains are real. So are the losses.

2026-04-15 8 min read

Vendor Lock-In Is Real

Durable Objects are a Cloudflare primitive. There is no equivalent in AWS Lambda, Google Cloud Run, or any other major cloud. The WebSocket hibernation API, the global uniqueness guarantee, the colocated durable storage—these are all Cloudflare-specific. If we ever needed to leave Cloudflare, the stateful layer would require a full rewrite.

This is not a hypothetical risk to dismiss. Cloudflare has changed pricing, deprecated features, and altered product roadmaps before. Betting the stateful core of a privacy application on a single proprietary primitive is a real exposure.

We made this bet anyway, because the Elixir branch on main remains a full, independent implementation that anyone can run without Cloudflare. The vendor dependency is in the hosted service, not in the protocol or the product itself. Anyone who needs infrastructure independence has a path.

No Persistent Background Processes

In Elixir, you can run a GenServer that holds state in memory across requests, acts on timers, and responds to messages from other processes. This is a powerful pattern. Our FxRate GenServer, for example, caches a live exchange rate in memory, refreshes it every 24 hours, and answers queries from any process in the system—all without a database round-trip.

In Workers, there is no persistent in-memory process. A Worker runs, handles a request, and exits. State must live in Durable Objects, D1, or KV. The FxRate equivalent on Workers is a KV cache with a Cron Trigger for refresh—more moving parts, more latency on cache miss. We built it: src/lib/fx_rate.ts writes the live rate to RATE_CACHE KV with a 25h TTL via a 0 6 * * * cron; paystack.ts reads it on checkout and applies the FX buffer. It works—it’s just more infrastructure than a single GenServer.start_link/1.

For sTELgano’s current workload, this is manageable. But the constraint is real: anything that benefits from long-lived in-memory state becomes more complex to implement correctly on Workers.

Debugging Is Harder

On a server running Elixir, debugging a production issue means connecting to the running node via iex --remsh, introspecting process state, tracing function calls live, inspecting the supervision tree, and reading structured logs with full request context. The BEAM was designed for hot observability of a running system.

On Workers, you have console.log and the Cloudflare real-time log tail. There is no equivalent of attaching to a running isolate and inspecting its state. Durable Object storage can be queried from the Cloudflare dashboard, but that’s a read-only snapshot, not a live session.

The practical impact: bugs that would take minutes to diagnose in a BEAM environment take longer on Workers. We compensate with more defensive logging and a comprehensive test suite. The local development environment (wrangler dev) runs a real Workers runtime, which helps catch most issues before they reach production.

The OTP Supervision Model

Erlang’s OTP supervision tree is one of the most robust fault-tolerance models in production software. Each process is supervised. A crash is contained to that process. The supervisor restarts it with a clean state. The rest of the system keeps running. This is not just theory—telecom systems built on the BEAM have achieved 99.9999999% uptime.

Cloudflare Workers has no equivalent. If a DO throws an unhandled exception, Cloudflare will restart it on the next request, but there is no supervisor that monitors it, no escalation path for repeated crashes, no fine-grained restart strategy. A crash in a DO during message handling will drop that message.

In practice, this is acceptable for sTELgano. A chat message that fails to send is visible to the user (the UI shows a delivery failure). The room is not corrupted—the DO’s durable storage is consistent. The user can retry. But anyone building on Workers should understand that the “let it crash” philosophy does not apply here in the same way. Unhandled errors need to be caught explicitly.

JavaScript vs Elixir: The Language Gap

Elixir has pattern matching, immutable data, the pipe operator, and a type system discipline (via Dialyzer) that catches a whole class of bugs at compile time. The language pushes you toward explicit error handling—every function returns {:ok, result} or {:error, reason}, and the compiler enforces that you handle both.

TypeScript with strict mode is a reasonable second. The type coverage on our Workers codebase is high, and Biome enforces style and catches common errors. But JavaScript’s implicit coercions, mutable state, and null/undefined dichotomy introduce a class of bugs that the BEAM simply doesn’t have.

We mitigated this with a comprehensive test suite: 75 tests across pure-function and real Workers runtime environments. But if language safety is your primary concern, Elixir on v1 is the more correct choice.

The Balance Sheet

Migrations are not free. The gains on v2 are real—zero infrastructure to manage, global deployment in seconds, no capacity planning, DO alarms instead of polling jobs. So are the losses—vendor lock-in, weaker fault isolation, harder debugging, no OTP supervision model.

We think the v2 tradeoffs are the right ones for a product that needs to reach users globally, deploy frequently, and keep a tiny team focused on building rather than operating. But we are honest about what we gave up. That honesty is the product.

Gained
  • +Zero infrastructure ops
  • +Global deployment in seconds
  • +Exact TTL expiry via DO alarms
  • +No capacity planning
  • +N=1 via serial execution (no locks)
Lost
  • −Infrastructure portability
  • −OTP supervision model
  • −Live process introspection
  • −Persistent in-memory processes
  • −Elixir’s compile-time safety
