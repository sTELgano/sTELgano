Self-Hosting sTELgano: Your Server, Your Rules

2026-03-08 8 min read

Why Self-Host?

The ultimate trust boundary. When you run your own sTELgano instance, you control the Worker code, the D1 database, and which secrets are in scope. You do not have to trust anyone — not even us.

Even though our hosted version is zero-knowledge by design, self-hosting removes the last remaining trust assumption: that we are actually running the code we publish. With your own Cloudflare account, you verify by deploying.

What You Need

Node.js 22+

The build toolchain runs on Node. Install via nvm, fnm, or your system package manager.

Cloudflare Account

Free tier is sufficient for personal use. Workers, D1, KV, and Durable Objects are all available on the free plan with generous limits.

Wrangler CLI

Comes with npm install. Wrangler is the single tool for local dev, D1 migrations, secret management, and deployment.

Domain Name

Optional but recommended. Cloudflare provides a free *.workers.dev subdomain out of the box if you prefer not to use a custom domain.

Provision Cloudflare Resources

After cloning and running npm install, create the two stateful resources Wrangler needs IDs for:

# D1 database — stores extension tokens and live counters
wrangler d1 create stelgano

# KV namespace — FX rate cache for currency conversion
wrangler kv:namespace create RATE_CACHE

Paste the returned IDs into wrangler.toml under [[d1_databases]] and [[kv_namespaces]]. The Durable Object and rate-limiter bindings are already declared and require no provisioning steps.

Secrets

Secrets are set once via Wrangler and never appear in your source code or wrangler.toml:

wrangler secret put ADMIN_PASSWORD
wrangler secret put ADMIN_USERNAME        # default: admin
wrangler secret put CF_AE_API_TOKEN       # Cloudflare Analytics Engine read token

If you enable monetization, also set the Paystack credentials:

wrangler secret put PAYSTACK_SECRET_KEY
wrangler secret put PAYSTACK_PUBLIC_KEY
wrangler secret put PAYSTACK_CALLBACK_URL
wrangler secret put PAYSTACK_RECEIPT_EMAIL_DOMAIN

Configuration Variables

Variable Required Purpose
PHX_HOSTYesYour production domain (e.g. stelgano.example.com)
CF_ACCOUNT_IDYesCloudflare account ID — visible in the dashboard URL
MONETIZATION_ENABLEDNoSet to true to enable paid tiers (default: false)
FREE_TTL_DAYSNoFree channel lifetime in days (default: 7)
PAID_TTL_DAYSNoPaid channel lifetime in days (default: 365)
PRICE_CENTSNoPrice in smallest currency unit (default: 200 = $2.00)
PAYMENT_CURRENCYNoISO 4217 display currency (default: USD)

These live in the [vars] section of wrangler.toml and are not secrets.

Custom Salts

The cryptographic salts (ROOM_SALT, ACCESS_SALT, SENDER_SALT, ENC_SALT) are public constants baked into the client JS at build time. They are domain separators, not secrets.

WARNING: Changing salts after users have created channels will make all existing channels permanently inaccessible. The same phone number and PIN will produce different hashes. There is no migration path. This is a one-way, destructive operation.

Only change salts before your first user creates a channel. If you are unsure, leave them at the defaults — they are safe to share publicly.

Deploy

Build the assets, apply D1 migrations, then deploy the Worker:

npm run build
wrangler d1 migrations apply stelgano --remote
wrangler deploy

That's the entire pipeline. Wrangler bundles the Worker, uploads static assets, and propagates to Cloudflare's global network. Within seconds the new code is live globally. Rollback is wrangler rollback.

Security Considerations

sTELgano ships with security defaults active out of the box:

  • HSTS — enforces HTTPS for all connections
  • CSP with per-request nonce — blocks inline scripts and external resources; no 'unsafe-inline' on script-src
  • Cache-Control: no-store — prevents caching of sensitive pages
  • Native CF rate limiters — 20/IP/min on /admin, 30/IP/min on WebSocket upgrades, 3/IP/min on new room creation

There is no server to patch. Cloudflare handles runtime security, TLS, and DDoS mitigation. Your operational surface is limited to keeping your API token and secrets rotated.

Maintenance

Essentially zero ongoing maintenance. Each room's Durable Object schedules its own alarm at creation time. When the alarm fires, the DO hard-deletes its state and broadcasts expiry to any connected clients. No cron jobs, no polling, no background queue to monitor.

  • Messages are hard-deleted immediately when a reply arrives (N=1 invariant)
  • TTL expiry fires exactly at the scheduled timestamp via DO alarms
  • The FX rate cache (if monetization is enabled) is refreshed daily at 06:00 UTC via a Cron Trigger

Upgrades

Pull, build, migrate, deploy:

git pull origin main
npm install
npm run build
wrangler d1 migrations apply stelgano --remote
wrangler deploy

Migrations are additive — they add tables and columns but never drop or rename existing ones. Your data is safe across upgrades.
