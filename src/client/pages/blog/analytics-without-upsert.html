Analytics Without UPSERT.

Why we replaced D1 UPSERT counters with Cloudflare Analytics Engine, how fire-and-forget events scale better than row locking, and how we kept country data out of room records.

2026-04-21 6 min read

The UPSERT Problem

The initial v2 port tracked metrics the same way v1 did: a country_metrics table and a daily_metrics table in D1, each incremented via INSERT ... ON CONFLICT DO UPDATE SET count = count + 1.

This works fine at low throughput. Under concurrent load, it becomes a bottleneck. SQLite UPSERT on a counter row acquires a write lock for the duration of the operation. Two concurrent room creates targeting the same (country, date) row will serialize. At burst load—many users creating rooms in the same country on the same day—these counter rows become a global lock point.

D1 is a distributed SQLite service. Write contention is exactly the kind of problem it’s not designed to absorb well. We needed a different model.

Cloudflare Analytics Engine

Cloudflare Analytics Engine (AE) is a time-series event store built into the Workers platform. You write events via writeDataPoint()—a fire-and-forget call that appends to an append-only log. There are no transactions, no locks, no rows to contend on. Each call is independent.

Aggregates are computed at query time via the Cloudflare GraphQL API, not stored as pre-computed counters. This shifts the cost: writes are cheap (no locking), reads are slightly more expensive (they aggregate on the fly). For our access pattern—many writes (every room create, every message), few reads (admin dashboard on demand)—this is the right trade.

Fire-and-forget means concurrent room creates never contend. A thousand rooms created simultaneously produce a thousand independent writeDataPoint calls. No serialisation, no lock waits, no latency spikes.

Our Event Schema

AE data points carry named blob and double fields. Our schema is minimal:

Field Value Notes
blobs[0] EventType room_free · room_paid · room_rejoin · room_expired_free · room_expired_paid · message_sent
blobs[1] country ISO Alpha-2 code derived client-side from the E.164 steg number (libphonenumber-js); always present for room_free, room_rejoin, room_paid, message_sent
blobs[2] CF-IPCountry Alpha-2 code from Cloudflare's IP geolocation; differs from blobs[1] for diaspora users, travellers, and VPN users
doubles[0] 1 Always 1 — count by SUM at query time

One data point covers both per-country and per-day breakdowns. Country is carried in the event itself. Date is implicit in the AE timestamp. The GraphQL query aggregates on both dimensions simultaneously.

Server-Blindness Preserved

A key constraint of the sTELgano privacy model: the server must never store a country code alongside a room hash. If it did, an attacker with database access could learn that a specific room was created from a specific country—narrowing the anonymity set for that room’s participants.

In v1, the rooms table had no country column. Country data lived in the aggregated country_metrics table, which held only totals per country per day—no per-room linkage.

In v2, the AE event carries a country code in blobs[1]. The rooms table in D1 still has no country column. The country is in the event stream alongside millions of other events, unlinked to any specific room hash. Room expiry events deliberately do not carry a country code—individual room records do not carry country metadata, so we cannot include it at expiry time either. Expiry events are global-only.

The country in an AE event is correlation-resistant by design: it appears alongside millions of other events with no room hash present. The room hash never appears in any AE event. The two datasets cannot be joined.

Reading the Data

The admin dashboard reads AE data via the Cloudflare GraphQL API at api.cloudflare.com/client/v4/graphql. The queryCountryMetrics() and queryDailyMetrics() functions in src/lib/analytics.ts issue authenticated GraphQL queries against the stelgano_eventsAdaptiveGroups dataset.

Both functions return [] gracefully when credentials are absent—a self-hoster who doesn’t configure CF_AE_API_TOKEN gets a dashboard with live counters but no historical charts. The live counters (active rooms) come from D1’s live_counters table, which is still maintained.

Diaspora Analysis

Because each event carries both a steg-number country (blobs[1]) and a CF-IPCountry (blobs[2]), a single GraphQL query grouped on both dimensions answers questions that neither dimension alone can:

  • → "How many rooms were created by Kenyan-number users connecting from outside Kenya?" (diaspora adoption)
  • → "Which countries have the largest gap between steg-number origin and physical connection?" (expat communities)
  • → "Do users in a given country tend to use local or foreign steg numbers?" (import vs. domestic usage)

The queryDiasporaMetrics() function in src/lib/analytics.ts issues this cross-dimension query and surfaces the results in the admin dashboard's Diaspora Breakdown table. Rows where the two country codes match represent domestic usage; mismatched rows are the diaspora signal.

No individual identity is at risk: the query returns aggregate counts per (steg_country, cf_country) pair. No room hash, no access hash, no phone number ever enters the AE event stream.
