One Room, One Thread.

How Cloudflare Durable Objects enforce the N=1 invariant without database-level locking, and why the actor model is the right primitive for private messaging.

2026-04-24 8 min read

The Concurrency Problem

The N=1 invariant—at most one message per room at any time—sounds simple. Implementing it correctly under concurrent load is not.

If two users send a message at the same moment, you need a guarantee that exactly one of their messages wins and the other is rejected. In a traditional relational database, you achieve this with a transaction: begin, delete old message, insert new one, commit. The database ensures serial execution through locking, and a UNIQUE constraint on the messages table as a second guard against any race that READ COMMITTED might permit.

That works. It’s also relatively expensive: two lock acquisitions, a write amplification under contention, and the overhead of coordinating with a remote database on every message send.

What Is a Durable Object?

A Durable Object is a stateful actor that lives on Cloudflare’s edge network. Each DO has a unique ID, its own durable SQLite storage, and a single-threaded JavaScript execution context. There is only one active instance of a given DO ID anywhere in the world at any moment.

That last property is the key one. Single-threaded serial execution means concurrent writes are structurally impossible. Not serialized by a lock—structurally impossible by the runtime model. Two requests arriving simultaneously are queued and processed one at a time, in order.

No transaction needed. No lock. The room’s DO processes one request at a time. The N=1 invariant falls out of the execution model for free.

One DO Per Room

Each sTELgano room is identified by a room_hash—a SHA-256 hex string derived from the steg number. When a user opens a channel, the Worker uses this hash as the DO ID. Cloudflare routes the request to a single DO instance, creating it if it doesn’t exist yet.

The DO stores the room’s state in its own SQLite: the current message (ciphertext + IV), the sender hash, the room tier, the TTL expiry timestamp, and the access hashes for the two participants. All of this state is colocated with the execution context. Reads are local, writes are local. No remote database round-trip for the hot path.

// Worker: route to room's DO
const id = env.ROOM_DO.idFromName(room_hash);
const stub = env.ROOM_DO.get(id);
const res = await stub.fetch(request);
// Inside the DO: single-threaded, no locking needed
async handleSendMessage(event) {
this.storage.delete("message");
this.storage.put("message", event.ciphertext);
}

Hibernatable WebSockets

A naïve WebSocket implementation on serverless infrastructure is expensive: the DO must stay fully active—executing JavaScript, holding in-memory state—for the entire duration of every open connection. For a privacy secure channel platform where users may keep a tab open for hours between messages, that’s a lot of idle compute.

Cloudflare’s Hibernatable WebSockets API solves this. When no messages are in flight, the DO can hibernate: the JavaScript context is evicted, but the WebSocket connections are maintained by the Cloudflare runtime. When a message arrives, the DO is woken up, its state is restored from durable storage, and the event is processed. From the client’s perspective, the connection never dropped.

The result: a room with two connected users who haven’t sent a message in an hour costs essentially nothing. The idle cost scales with active rooms, not connected rooms.

Alarms for TTL Expiry

In v1, an Oban job ran hourly to find and expire rooms whose TTL had passed. In v2, each room’s DO schedules its own alarm—a timer that wakes the DO at a specific timestamp. When the alarm fires, the DO hard-deletes its state, broadcasts the expiry event to any connected WebSocket clients, and marks itself as expired.

This is cleaner than a polling job. Expiry is exact (not hourly ± runtime jitter), colocated with the room’s own execution context, and requires no external job queue infrastructure. The alarm is stored durably—if the DO hibernates between now and the alarm time, Cloudflare will still fire it.

Comparison: BEAM Processes vs Durable Objects

BEAM Process (v1)

  • One process per WebSocket connection
  • Supervised by OTP supervisor tree
  • In-memory state, lost on crash (supervisor restarts)
  • PostgreSQL as shared state store
  • Transactions for N=1 enforcement
  • Oban for TTL expiry scheduling

Durable Object (v2)

  • One DO per room (all connections share it)
  • Managed by Cloudflare runtime
  • Durable SQLite storage, survives restarts
  • Local storage, no remote DB for hot path
  • Serial execution for N=1 (no locks needed)
  • DO alarms for exact TTL expiry

Neither model is universally better. The BEAM’s process isolation is more granular—a bug in one connection does not affect others. Durable Objects give you durable storage colocated with execution, which eliminates a class of distributed-state bugs entirely. For sTELgano’s workload (one room = one unit of state), the DO model is an exceptionally clean fit.
