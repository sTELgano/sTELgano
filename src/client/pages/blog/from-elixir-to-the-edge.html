From BEAM to the Edge.

Why we rewrote sTELgano in TypeScript on Cloudflare Workers, what prompted the move, what stayed the same, and what happens to the Elixir branch.

2026-04-27 10 min read

Why Elixir Was the Right Start

We wrote an entire post explaining why we chose Elixir for the first version of sTELgano. The short version: the BEAM VM was literally designed for soft real-time communication systems, Phoenix Channels gave us 2M+ concurrent WebSockets on a single node, and the supervision tree model meant production crashes were contained and self-healing.

Elixir was right for v1. We stand by that. A small team building a real-time privacy application for the first time, where correctness mattered more than operational simplicity, reached for the best-in-class tool for the job. The BEAM's lightweight processes, each isolated and supervised, gave us exactly the fault model we needed.

But after running v1 in production, a different set of constraints became visible.

The Operational Reality

Running an Elixir application in production on a bare server means owning the full stack: OS patching, Erlang/OTP upgrades, release management, database migrations, systemd unit configuration, nginx TLS termination, Let’s Encrypt renewal, Oban job queues, and a monitoring setup to know when any of these go wrong.

For a product with a clear privacy thesis and a small team, this is non-trivial overhead. Each layer is a surface to maintain and a potential point of failure. A missed security patch in the OS is as dangerous as a bug in the application code. Capacity planning for a single-region deployment means either over-provisioning or accepting degraded performance under burst load.

We wanted to spend engineering time on the product—on the cryptography, the UX, the protocol—not on keeping a server healthy.

Why Cloudflare Workers?

Cloudflare Workers runs JavaScript on V8 isolates at 330+ points of presence globally. There are no servers to manage, no capacity decisions to make, no patching cycles. Deployment is wrangler deploy and the code is live everywhere in seconds.

But the primitive that made Workers the right choice for sTELgano specifically is the Durable Object. A DO is a stateful, single-threaded actor that lives at the edge. Each sTELgano room gets exactly one DO instance. That single-threaded execution model is what makes the N=1 invariant implementable without database-level transactions—serial execution means concurrent writes are structurally impossible.

Phoenix Channels gave us one BEAM process per room. Durable Objects give us one V8 isolate per room. The model is the same; the infrastructure underneath is completely different.

What Did Not Change

The sTELgano-std-1 protocol is unchanged. The cryptographic constants (ROOM_SALT, ACCESS_SALT, SENDER_SALT, ENC_SALT), the derivation functions, the AES-256-GCM parameters, the 600,000 PBKDF2 iterations—all identical. Existing rooms created on v1 are accessible on v2 without any migration.

Protocol

sTELgano-std-1 unchanged. All crypto constants and derivation paths identical.

N=1 Invariant

At most one message per room. Reply deletes previous atomically. Same guarantee, different mechanism.

Privacy Model

Server sees only hashes and ciphertext. No phone number. No PIN. No plaintext. Unchanged.

The Future of the Elixir Branch

The Elixir implementation lives on the main branch and is not going anywhere. It is a complete, production-grade implementation of sTELgano-std-1 and a useful reference for anyone who wants to self-host on their own infrastructure without Cloudflare.

If you run an Elixir stack, want full infrastructure control, or operate in a jurisdiction where edge-computing providers are not an option, the Elixir branch is the right choice. It will continue to receive security patches and protocol-level updates whenever sTELgano-std-1 changes.

Both branches are AGPL-3.0. Both implement the same protocol. New features may land in the Cloudflare branch first, but anything that affects the protocol or the cryptographic spec propagates to both.

main — Elixir/Phoenix. Self-hosted on your own server. Full infrastructure control.
v2-cloudflare — TypeScript. Deploy to Cloudflare Workers with wrangler deploy.

The Tradeoff We Made

This migration was not free. Cloudflare Workers is a proprietary platform. Durable Objects are not a portable abstraction—if we ever wanted to move off Cloudflare, we would need to rewrite the stateful layer. We traded infrastructure autonomy for operational simplicity and global reach.

We think that’s the right trade for v2. The product’s privacy guarantees live in the protocol and the client-side cryptography, not in who operates the server. A Cloudflare-hosted sTELgano room is as private as a self-hosted one—the server sees the same hashes and ciphertext either way. The trust model is in the math, not the infrastructure provider.

We cover the full set of tradeoffs in a dedicated post. This is an honest product. We do not paper over the downsides.
